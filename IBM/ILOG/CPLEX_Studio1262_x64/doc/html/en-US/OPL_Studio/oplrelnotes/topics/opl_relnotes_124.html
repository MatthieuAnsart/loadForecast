Release notes for CPLEX Studio V12.4

Summarizes new and improved features in CPLEX® Studio.

Improvements to the SPSS Connector

  • More SPSS® data streams are accepted as input from SPSS Modeler.

  • Non-alphanumeric characters ($, &, #, %, -, ., or an empty string) in node names are accepted by OPL. They are automatically replaced with an underscore (_).

  • In the IDE, streams are represented graphically.

    When you right-click on a stream and select Open With > SPSS Stream Editor, the nodes of the stream are represented graphically. The stream editor keeps the same layout as SPSS Modeler.

    If several nodes have the same name, the names of those nodes are displayed in red.

See also the tutorial IBM SPSS Connector in CPLEX Studio.

Scheduling improvements

  • New feature: deterministic parallel search

    The multi-worker mode in CP Optimizer now follows a deterministic behavior. When the same random seed, the same number of workers and the same deterministic (time independent) limit, such as a branch or fail limit, are specified across multiple runs, CP Optimizer delivers the same solutions. Note that, although logs produced by such runs may differ at some point, depending on how workers are individually scheduled by the system, individual worker behaviors are the same, and solutions delivered by the parallel run are the same.

  • Element expressions

    Floating-point element expressions are now supported in CP Optimizer.

    Two limitations that existed in previous versions of CP Optimizer no longer exist.

    • The first limitation concerned the type of elements in a table.

      IloNumExprArg IloNumArray::operator[] (IloIntExprArg intExp) const

      The operator [ ], which was already available, worked only when the values in the invoking array were integers, despite the fact that it was declared as a Num array. This operator now works with floating-point values.

    • The second limitation concerned the type of the indices in multidimensional tables.

      In an expression T[index1][index2]...[indexN], if one indexI was a decisionExpr, all the indexI had to be integer expressions. If indexI contained a decisionExpr, it was possible to use any kind of usual map index (integer, string, tuple, ...).

      This limitation no longer exists.

      Now, in an expression T[index1][index2]...[indexN], if one indexI is, or several indexI are, decisionExpr, only these decisionExpr must be of the type integer (dvar int or dexpr int). The other constant indices can be of any usual index type (int, float, string, tuple).

      For example:

      float distance[locations][locations] = ...;
dvar int deliveries[trucks][visits] in locations;
minimize
   sum(t in trucks, v in visits: v > 1)
   distance[deliveries[t][v-1]][deliveries[t][v]];

    For details, see Element constraints
