OPL 6.x concepts - project files vs. folders

OPL projects are handled differently in OPL 6.x, compared to previous releases.

There are some important concepts to understand before importing projects created with previous versions of OPL into the OPL IDE.

How projects were defined previous to OPL 6.x:

  • OPL 5.x (and previous releases) used the concept of a project to associate a model (.mod) file with one or more data (.dat) files. From OPL 5.x onwards, there could also be one or more settings (.ops) files, and one or more run configurations.

  • The relationships between these elements was maintained by a project (.prj) file.

  • The project_name.prj file could be double-clicked in Windows Explorer to launch that project in the OPL IDE.

How projects are defined in OPL 6.x:

  • In OPL 6.x, the same project elements are present and are still referred to as a project, but they are stored in named folders under a root folder.

  • All of the elements of the OPL project — model files, data files, settings files and run configurations — must be stored in the same folder or one of its subfolders.

  • Project (.prj) files no longer exist.

  • Instead, a .project file stores information specific to the OPL IDE environment, and an .oplproject file stores information similar to what was stored in the old .prj files.

  • Projects created with previous releases (OPL 3.x, OPL 4.x, and OPL 5.x) must be migrated and converted to the OPL 6.x format before they can be used.

    • Projects created in OPL 5.x can be migrated directly using the OPL 6.x Import Wizard.

    • Because of language changes between versions, projects created in OPL 3.x or OPL 4.x should be re-created as new projects in OPL 6.x.
