Types and declarations

Lists the changes to types and declarations in OPL 4.

Parent topic: The modeling language

Enumerated values

The type enum is not supported in OPL 4. Use set of strings {string} instead.

In OPL 3

You wrote in the model file:

enum Components ...;

and in the data file:

Components = { nitrogen, hydrogen, chlorine };

In OPL 4

You write:

{string} Components = ...;
Components = { "nitrogen", "hydrogen", "chlorine" };

and in the data file:

Components = { "nitrogen" "hydrogen" "chlorine" };

Note that the comma “,” between the items is no longer required in the data file, though it does work in order to provide greater compatibility with OPL 3.

In OPL 3

You wrote:

enum Components { nitrogen, hydrogen, chlorine };

In OPL 4

You write in the model file:

{string} Components = { "nitrogen", "hydrogen", "chlorine" };

Ranges

A range is now a value and no longer a type.

In OPL 3

It was possible to use a range as a type, for example:

range Workers 1..nbWorkers;
{ Workers } qualified[Tasks] = ...;
range Obj 0..(n*n);
var Obj obj;

In OPL 4

You must use one of the standard types when declaring data or a decision variable, as in:

range Workers = 1..nbWorkers;
{int} qualified[Tasks] = ...;
range Obj = 0..(n*n);
dvar int obj in Obj;

float+, int+

Use these types for decision variables only. They can be seen as shortcuts for domain specifications. For example:

dvar float+ myVar;

is equivalent to 

dvar float myVar in 0..infinity;

float+ and int+ are no longer supported  for data declaration. Use assert statements to check the domains of data.

In OPL 3

It was possible to declare:

int+ nonnegArray[1..10]= ...;

and the system raised an error if an initialization with a negative value is attempted.

In OPL 4

The corresponding declaration is:

int nonnegArray[1..10]= ...;

and the test can be done as

assert forall (i in 1..10) nonnegArray[i] >= 0;

Boolean variables

The keyword boolean can be used as a shortcut to Boolean variables. For example:

dvar boolean myVar;

is short for 

dvar int myVar in 0..1;

Literal strings

Backslash '\' in string literals needs to be escaped, in model and data files.

In OPL 3

You wrote:

... "c:\mydata\manufact.dat" ...

In OPL 4

You write:

... "c:\\mydata\\manufact.dat" ...

End of line within a string literal needs to be escaped.

In OPL 3

You wrote:

... "SELECT a FROM table
     WHERE ... "

In OPL 4

You write:

... "SELECT a FROM table \
     WHERE ... "

Arrays

Multidimensional arrays are now arrays of arrays. Replace [i,j] with [i][j] .

In OPL 3

You wrote:

[i,j] 
int myarray[R, 1..5]; 
myarray[2, 3];

In OPL 4

You write:

[i][j] 
int myarray[R][1..5]; 
myarray[2][3];

When declaring generated arrays using the compatibility syntax [,] all the dimensions need to be qualifiers. For example:

In OPL 3, it was possible to initialize a multidimensional array like this:

int myArray[i in R,j in S] = i;

In OPL 4, this syntax remains valid for compatibility reasons, but the exact syntax is:

int myArray[i in R][S] = i;

Note that the following code is not correct:

int myArray[i in R,S] = i;

Grouped declarations

ILOG OPL Development Studio supports single-identifier declarations only. Comma-separated declarations for identifiers of the same type are no longer supported. For example, replace

int a,b;

with

int a;
int b;

Conversion from a range to a set of integers

In OPL 3

You could initialize an integer set from a range, like this: 

{int} intSet = 1..13;

In OPL 4

You need to use the asSet method explicitly:

{int} intSet = asSet(1..13);

because OPL 4 does not support implicit type conversions.
