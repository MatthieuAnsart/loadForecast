Deterministic time

Announces new routines and methods to measure time deterministically.

Perhaps you recall that CPLEX V12.4 introduced the concept of deterministic time; that is, a measure of time that respects the same solution path to arrive at the same values in the solution while it yields the same level of performance for repeated solving of the same model with the same parameter settings, on the same computing platform. Now, CPLEX V12.5 extends that idea with methods and routines that support deterministic time stamps. By default, deterministic time is now reported as ticks in the CPLEX log, along with conventional time in seconds. Deterministic time stamps are available from callbacks and during the search for a solution, during probing, during solution polishing, during tuning. For an introduction to this extended idea of deterministic timing, see the topic Determinism and the timing interface in the CPLEX User's Manual.

For a list of parameters governing deterministic time in probing, solution polishing, and tuning, see the release note New parameters. The table in that note includes links to the documentation of each of those new parameters.

CPLEX V12.5 also offers examples of these new deterministic routines and methods in use.

  • In the Callable Library (C API), see the examples xmipex4.c and mipex4.c.
  • In the Python API, see the example mipex4.py.
  • In the C++ API, see the example ilomipex4.cpp.
  • In the MATLAB connector, see the example cplexgui.m.

This extension of deterministic timing is also available in the Interactive Optimizer. Help for the new deterministic commands is available on line in an Interactive session.

From callbacks, these two additional items of information are now available: CPX_CALLBACK_INFO_STARTTIME and CPX_CALLBACK_INFO_STARTDETTIME.
