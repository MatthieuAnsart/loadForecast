Callbacks, threads, and parallel execution

Behavior of parallel execution in the presence of multiple threads and callbacks now conforms to documentation.

This conversion note highlights changes in CPLEX V12.5.1 that can impact existing applications using threads and callbacks in deterministic parallel execution.

Corrected behavior

Users who set the threads parameter (global thread count) to a value strictly greater than 1 (one) in order to invoke deterministic parallel optimization in the presence of control callbacks will observe that CPLEX invokes a different clone of the callback object for each thread now (as documented).

Furthermore, the methods IloCplex::MIPCallbackInfoI::getMyThreadNum and IloCplex::MIPCallbackInfoI::getUserThreads now return the correct index identifying the thread currently executing; that is, these methods return consistent data now (as documented).

Context

In previous releases of CPLEX, the method IloCplex::MIPCallbackInfoI::getMyThreadNum to access the index identifying the thread currently executing in an application of multiple threads in deterministic parallel always returned 0 (zero) and the method IloCplex::MIPCallbackInfoI::getUserThreads always returned 1 (one) in the presence of a control callback, while the same callback was being invoked on all threads. This anomaly has been corrected in CPLEX V12.5.1.
